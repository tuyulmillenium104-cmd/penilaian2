## 🚫 ANTI-TEMPLATE RULES (NEW IN V3.5)

### Rule 1: NO Generic Problem Statements

| ❌ TEMPLATE (Avoid) | ✅ ORIGINAL (Use) |
|--------------------|-------------------|
| "Banks can't use public blockchains" | Personal story: "Got into a debate last week..." |
| "Institutions face a dilemma" | Personal realization: "Vitalik's point keeps coming back to me" |
| "Privacy is essential for institutions" | Contrarian take: "Here's what actually matters..." |
| "The challenge with public chains" | Specific scenario: "What I've seen working with compliance teams..." |

### Rule 2: NO Feature Lists in Bullet Format

| ❌ TEMPLATE (Avoid) | ✅ ORIGINAL (Use) |
|--------------------|-------------------|
| "Features include: a) role-based permissioning, b) Proxy RPC, c) selective disclosure" | ONE feature with insight: "Proxy RPC isn't a wrapper - it's how regulated entities actually operate" |
| "Key capabilities: privacy, interoperability, compliance" | Natural flow: "Private execution. Public settlement. That's the combo." |

### Rule 3: NO Mandated Phrases as Conclusion

| ❌ TEMPLATE (Avoid) | ✅ ORIGINAL (Use) |
|--------------------|-------------------|
| "...forming The Bank Stack of Ethereum" (at the end) | Use mandated phrase EARLY or MID-thread, not as conclusion |
| Ending with branding statement | End with conversational close: "Anyway, the mental model shift is real" |

### Rule 4: NO Standard Terminology Without Twist

| ❌ TEMPLATE (Avoid) | ✅ ORIGINAL (Use) |
|--------------------|-------------------|
| "Validium architecture keeps transaction data off-chain" | "Banks don't broadcast sensitive data. They settle proofs." |
| "State roots and ZK proofs provide verification" | "Only state roots and ZK proofs hit Ethereum" (casual) |
| "Native interoperability via Elastic Network" | "No third-party bridges (we've seen how those go)" - adds personality |
